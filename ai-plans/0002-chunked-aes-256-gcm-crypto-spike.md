## Rationale

Validate the core encryption format before building upload and download workflows on top of it. ShadowDrop depends on chunked AES-256-GCM so encrypted files can still support resumable range downloads.

## Acceptance Criteria

- [x] A shared crypto component can encrypt plaintext into independently decryptable AES-256-GCM chunks.
- [x] A shared crypto component can decrypt full files from encrypted chunks.
- [x] A shared crypto component can decrypt only the chunks needed for a plaintext byte range.
- [x] One share-level encryption secret with at least 256 bits of randomness can be used to derive per-file AES-256-GCM content keys.
- [x] Per-file AES-256-GCM content keys are derived with HKDF-SHA-256 using the share secret, a non-secret per-share KDF salt, and file-specific context.
- [x] Each chunk uses a unique 96-bit nonce under its derived file content key.
- [x] Chunk authentication includes relevant metadata such as encryption version, algorithm, share id, file id, chunk size, chunk index, and plaintext chunk length.
- [x] Tampered ciphertext, tampered metadata, mismatched nonce metadata, wrong share secrets, and wrong file derivation context fail decryption.
- [x] The encryption format includes an algorithm/version marker so future formats can coexist.
- [x] Automated tests cover full-file round trips, range-oriented decryption, mid-chunk ranges, multi-chunk ranges, and failure cases.
- [x] The shared crypto code remains compatible with Native AOT usage from the CLI.

## Technical Details

Implement this spike in `ShadowDrop.Shared` because both the API and CLI need to understand the encrypted file format. Keep the public API small and focused around encrypting chunks, decrypting chunks, and mapping plaintext byte ranges to chunk ranges.

Use AES-256-GCM for the MVP format. A share should have one random share-level encryption secret generated by the uploading client. The share secret should contain at least 256 bits of randomness. Recipients should only need this one secret for the whole share, even when the share contains multiple files.

Do not directly reuse the same AES-GCM content key across every file. Derive a per-file 256-bit AES-GCM content key from the share-level encryption secret with HKDF-SHA-256. Use the share encryption secret as the input keying material, a non-secret per-share KDF salt stored with share metadata, and file-specific info that includes the encryption format version, algorithm identifier, share id, and file id. The HKDF output length should be 32 bytes. This keeps the recipient experience to one share secret while reducing the risk of nonce namespace mistakes across files.

Each chunk must have a unique 96-bit nonce for its derived file content key. The nonce strategy should be deterministic from safe inputs or explicitly stored, but it must make nonce reuse under the same derived key impossible in normal operation. The chosen nonce format should be documented in the implementation and covered by tests.

The authenticated data for every chunk should bind the chunk to its context. Include at least the encryption format version, algorithm identifier, share id, file id, configured chunk size, chunk index, and plaintext chunk length. This prevents valid encrypted chunks from being silently replayed in the wrong position or under the wrong metadata.

Range-oriented tests should include ranges that start in the middle of a chunk, end in the middle of a chunk, align exactly to chunk boundaries, and span multiple chunks.

The implementation should not require reading an entire file into memory. Tests can use in-memory byte arrays, but the design should be compatible with later stream-based upload and download pipelines.

Because the CLI must support Native AOT, the shared crypto implementation should avoid runtime code generation, dynamic loading, and reflection-heavy patterns that could break trimming or AOT publishing.
