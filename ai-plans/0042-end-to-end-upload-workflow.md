## Rationale

Turn the non-interactive CLI into a complete sender workflow instead of exposing the API's upload-intake boundary. A user who supplies files to `upload` should receive a usable share URL and the non-retrievable credentials required to download its files, and may optionally create a recipient queue file in the same invocation. Lower-level upload, share-creation, and queue-generation commands remain available for scripting, composition, and recovery. This work also removes the misleading use of `share-id` where the public download capability is actually an opaque share token or URL.

## Acceptance Criteria

- [ ] `shadowdrop upload <files...>` encrypts and uploads all files with one share key, creates one share only when every requested upload succeeds, reports the public share URL, and delivers every non-retrievable credential required for download.
- [ ] The end-to-end upload command exposes documented share options for expiration, direct-HTTP versus separate-key mode, and optional download bearer-token generation. Defaults are explicit and consistent with API validation: the default expiration is 7 days, the default sharing mode is separate-key, and download bearer-token generation is off by default (when generated, the bearer token's expiration defaults to the share expiration). Invalid option combinations (for example direct-HTTP combined with bearer-token generation) are rejected during CLI validation.
- [ ] `upload --queue-out <path>` writes a valid download queue after share creation and therefore requires no separate share-creation flag; like `--secrets-out`, it refuses to overwrite an existing file unless `--force` is supplied.
- [ ] Generated queue entries use the created share's public token and manifest metadata and assign sanitized collision-safe relative output paths. Queues contain no secrets by default, while an explicit `--embed-secrets` option can create a self-contained sensitive queue containing the share key and any required download bearer token.
- [ ] A lower-level `upload raw` command preserves encrypted intake for scripts, reports uploaded file IDs, and always delivers its non-retrievable share key through the selected secret destination.
- [ ] A lower-level share-creation command can create a share from previously uploaded file IDs and supports the same share options as the end-to-end upload workflow.
- [ ] A lower-level queue-generation command can fetch an existing share manifest by public share token or URL and write a queue file, including support for separately supplied credentials and optional explicit embedding of those credentials.
- [ ] Public CLI terminology consistently distinguishes the internal share UUID from the public share token or URL, including download arguments, help text, structured output, and queue fields.
- [ ] Any breaking queue-format change increments the queue format version (`FormatConstants.QueueVersion`), and incompatible versions are rejected with a clear error.
- [ ] By default, a successful end-to-end upload writes the public share URL, plaintext share key, and any generated download bearer token to stdout exactly once. The existing `--output-secret` opt-in is removed because a share without delivered credentials is unusable.
- [ ] `--secrets-out <path>` writes all non-retrievable download credentials to a dedicated file instead of stdout. The file is written atomically, refuses to overwrite an existing file unless `--force` is supplied, and uses owner-only permissions on supported platforms with documented limitations elsewhere.
- [ ] An explicit JSON output mode provides a stable, Native-AOT-compatible schema containing uploaded file IDs, internal share ID, public share token and URL, generated artifact paths, operation status, and the required credentials unless they were redirected to `--secrets-out`, in which case it reports the credentials-file path instead.
- [ ] A command that creates a share cannot report success unless its public share capability, share key, and generated download bearer token have all been delivered through the configured output channel.
- [ ] Secret-free queues continue to require credentials through separate download inputs. Self-contained queues can be downloaded without separate credential arguments, and conflicting embedded and explicit credential sources are rejected rather than resolved ambiguously.
- [ ] Plaintext share keys, generated download bearer tokens, and upload tokens never appear in stderr, diagnostics, or logs. Queue embedding occurs only with `--embed-secrets`, and secret-free queues never serialize credential properties.
- [ ] Complete success, partial upload failure, share-creation failure, and queue-write failure have deterministic stdout, stderr, JSON, and exit-code behavior. Successful upload IDs and any created share remain identifiable for recovery, but the overall command never reports success when a requested stage fails.
- [ ] Queue files are written atomically so validation, serialization, cancellation, or local I/O failure cannot leave a completed-looking partial file. Queues containing embedded secrets receive the same restrictive file handling as dedicated credential files.
- [ ] Interactive upload and download reuse the same upload, share-creation, queue-building, validation, and result models as non-interactive commands rather than maintaining separate orchestration logic.
- [ ] User-facing documentation and command help describe the primary end-to-end workflow, secret delivery, generated queue usage, lower-level scripting/recovery commands, and migration from the previous intake-only `upload` behavior.
- [ ] Existing bounded-memory encryption/upload streaming, zero-knowledge key handling, generic error-message rules, configuration precedence, and Native AOT compatibility remain intact.
- [ ] Automated tests cover the multi-file end-to-end upload path, mandatory credential delivery, and the stdout/file/JSON credential destinations.
- [ ] Automated tests cover self-contained and secret-free queues, file-name collision handling, and restrictive/atomic file handling for queues and credential files.
- [ ] Automated tests cover each lower-level command (`upload raw`, `share create`, `queue create`), option validation and implications, and failure/recovery results.
- [ ] Automated tests cover terminology and help text, interactive delegation to the shared orchestration, and secret non-leakage from unauthorized channels (stderr, diagnostics, logs).

## Technical Details

Refactor the CLI around reusable application operations rather than having command handlers call one another. Separate operations should own upload intake, share creation, manifest retrieval, queue construction, and atomic queue persistence. The end-to-end `upload` handler composes those operations in order; `upload raw`, `share create`, and `queue create` expose the same operations as lower-level command surfaces. Interactive handlers should collect choices and render results around this shared orchestration.

Treat `upload` as a pre-MVP contract replacement, not as an additive `--create-share` mode. `--queue-out` is valid only on the end-to-end path and naturally runs after share creation. Default to a 7-day expiration and separate-key sharing mode (the API's natural default and the zero-knowledge-friendly choice), with download bearer-token generation off by default; document these and allow explicit overrides. Keep mutually exclusive or dependent options enforced during CLI validation before file I/O or network requests begin. The lower-level share command must accept file IDs in a deterministic order and must not require the plaintext share key because the server does not receive that key.

Replace `--output-secret` with output-destination semantics. Default text output writes labeled share URL, share-key, and generated bearer-token values to stdout exactly once; all diagnostics remain on stderr. `--secrets-out` redirects the credential values to a source-generated JSON credential document and leaves only its path and non-secret result data in normal output. Prepare and validate the selected sink before uploads begin, retain credential material until the sink has completed successfully, and return non-zero if delivery fails. Because local output and remote share creation cannot form one transaction, minimize the failure window through atomic local writes and report any resulting remote state without claiming success.

Introduce a shared command-result model and source-generated JSON contracts for command results and credential files. Human output should lead with the public share URL and clearly separate identifiers, artifact paths, and secret material. JSON output should use named fields rather than an ordered line protocol and should represent completed stages and recoverable identifiers on failure. It includes the required credentials by default because selecting JSON is itself an output destination; when `--secrets-out` is present, the result contains the credential document path instead. Preserve non-zero exit status whenever any requested stage fails. In particular, do not create a share after a partial upload, and report a queue-write failure without hiding the already-created share or its already-delivered credentials.

Build queues from normalized share-manifest entries through one queue builder used by both end-to-end upload and `queue create`. Use the public share token consistently instead of storing it in a property named `shareId`; because this changes the serialized contract, increment `FormatConstants.QueueVersion` and keep strict version validation. Model optional credentials once at queue/share scope rather than duplicating them in every file entry. Omit the credential object entirely for normal secret-free queues. `--embed-secrets` requires a queue output and all credentials needed by that share, marks the output as sensitive, and allows the download command to resolve credentials from the queue after rejecting conflicts with explicit CLI inputs.

Derive default output paths from sanitized leaf file names, resolve duplicate names deterministically without introducing directories, and reject entries that cannot produce a safe name. Queue and credential output should be written to temporary sibling files, flushed, assigned restrictive permissions where supported, and atomically moved into place. Refuse to overwrite existing queue or credential files unless `--force` is supplied, rather than silently replacing them, and clean up temporary files after failures or cancellation.

Treat a queue created with `--embed-secrets` as a complete download capability rather than as an ordinary queue. Document that anyone possessing it can download the share and that embedding removes the security benefit of delivering the public share reference and credentials through separate channels. Never embed the upload authorization token. Tests must capture every output channel and generated artifact and assert that credentials occur only in the configured destination, while still proving that every successful share-creating command delivered them somewhere.

Scope is limited to `ShadowDrop.Cli` and `ShadowDrop.Shared`; `ShadowDrop.Api` is not modified unless a required capability is genuinely missing, in which case the necessary API change is added deliberately rather than worked around. This work is large and is best landed as a stacked sequence of PRs: (a) terminology rename distinguishing the internal share UUID from the public token, plus the `FormatConstants.QueueVersion` bump and version rejection; (b) the end-to-end `upload` workflow with mandatory secret delivery, `--secrets-out`, and JSON output; (c) the lower-level `upload raw`, `share create`, and `queue create` surfaces plus `--embed-secrets`; (d) interactive delegation to the shared orchestration and user-facing documentation.
