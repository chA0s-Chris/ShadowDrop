# Polish CLI help and version output behavior

> Issue: [#92](https://github.com/chA0s-Chris/ShadowDrop/issues/92)

## Rationale

The CLI currently documents global version/help behavior that is not honored by the actual root-command dispatch path. `./ShadowDrop.Cli --version` is treated like a missing command, invoking the executable without arguments prints only an error, and the help aliases include the unusual `-?` alias. The command-line surface should behave like a conventional CLI: `--version` should report the installed build version, no-argument invocation should show help, help should be available through `-h` and `--help`, help should start with a simple product/version header, and the decorated product/version banner should be visible at the start of command output unless explicitly suppressed for scripting or routed away from machine-readable output.

## Acceptance Criteria

- [x] `./ShadowDrop.Cli --version` prints only `ShadowDrop v<version>` (e.g. `ShadowDrop v1.0.21`) and exits successfully, without printing the decorated banner.
- [x] `./ShadowDrop.Cli` with no parameters displays help instead of an error-only message.
- [x] `-?` is no longer advertised or accepted as a help alias.
- [x] `-h` and `--help` continue to display help.
- [x] Help output begins with a simple `ShadowDrop v<version>` header, using the actual build/version value, followed by a blank line and the generated help text.
- [x] Command output begins with the decorated two-line `ShadowDrop v<version>` banner, using the actual build/version value, unless one of the explicit suppression/corruption-avoidance rules applies.
- [x] A global `--no-banner` option suppresses the decorated banner for command output; help output does not use the decorated banner and continues to show its simple version header.
- [x] The banner never corrupts machine-readable stdout, script-style stdout contracts, or downloaded bytes; for byte-stream, JSON-producing, credentials-only, `file-id:`, `share-key:`, `share-url:`, `queue-file:`, `share-revoked:`, `share-cleanup:`, or similar parseable output modes, the banner is written to stderr or suppressed according to the command's output contract.
- [x] The banner's second-line hyphen run expands when the actual `v<version>` text is longer than `vx.x.x`, keeping the top and bottom lines the same width.
- [x] When the output terminal supports color, the banner uses `dodgerblue1` border/hyphen styling, `cyan1` slash styling, bold white `ShadowDrop ` text, bold `dodgerblue1` `v` text, and bold `cyan1` version text.
- [x] Automated tests cover the updated version, help, alias, banner shape, banner padding, `--no-banner`, plain `--version`, colored banner rendering, corruption-avoidance routing, and non-colored fallback behavior.

## Technical Details

Keep the change centered on `src/ShadowDrop.Cli/CliApplication.cs`, where the root `System.CommandLine` model is created and parsed before the manual command dispatch in `ExecuteAsync`. The current `HelpAction` fast path already routes help output through `services.StandardOut`; extend this area so root help can be invoked for an empty argument list and so `--version` is handled before the missing-command path writes `A command is required.`. Use the assembly informational version generated by the existing build pipeline (`SetInformationalVersion`) rather than introducing a separate version constant. `--version` should remain plain machine-readable output and should not go through the decorated banner renderer; it must print `ShadowDrop v<version>`, e.g. `ShadowDrop v1.0.21`. `System.CommandLine`'s default `VersionOption` only emits the bare version string, so replace or override its action with one that prints the `ShadowDrop v<version>` format instead. The decorated banner should
have a single shared formatter used by command execution paths so those surfaces cannot drift.

The plain, non-colored banner shape is:

```text
.--// ShadowDrop vx.x.x \\--.
`--/---------------------\--´
```

For an actual version, replace only the `x.x.x` placeholder with the version number while keeping the literal `v` in `ShadowDrop v`. The second-line hyphen count should be derived from the rendered top line so both lines stay the same width. With the sample above, `ShadowDrop vx.x.x` uses 21 hyphens on the second line; a longer version such as `ShadowDrop v1.0.21` needs one additional hyphen.

Add a global `--no-banner` option to the root command and make it available consistently to subcommands. This option exists for scripting and other output-sensitive uses; when present, command execution should skip the decorated banner entirely while leaving the rest of the requested output unchanged. Help output is already non-decorated and continues to show its simple `ShadowDrop v<version>` header even when `--no-banner` is present. `--no-banner` should not be required for `--version`, because `--version` never prints the banner.

Adjust the `System.CommandLine` configuration or root command setup so help aliases are limited to `-h` and `--help`. Verify that `-?` is neither displayed in generated help nor accepted as a valid help request. Preserve the existing `--` separator behavior: operands after `--` that look like help flags or `--no-banner` must remain command operands, while help flags before `--` still trigger help.

For help output, prepend a simple `ShadowDrop v<version>` line and a blank line before the generated `System.CommandLine` help text. This keeps help visually conventional while still exposing the installed version at the top. For command execution, do **not** centralize banner emission in `CliApplication.ExecuteAsync` before dispatching to a handler — a handler's early-return validation/precondition paths (e.g. `ShareRevokeCommandHandler`'s invalid-share-id check, configuration-resolution failures, API exceptions) are only distinguishable from its success path once the handler has actually run, so a banner written up front cannot be un-written if the handler turns out to fail with an error-only outcome. Instead, pass the shared banner formatter and the resolved `--no-banner` flag into each command handler's constructor. Each handler calls the formatter itself, immediately before its first real (non-error) output, once its own validation/precondition checks have passed. Handlers that
return early with only an error message never call the formatter, so no banner is printed for parse errors, validation
errors, configuration/TLS failures, or handler failures that only emit errors, without needing any central suppression or buffering logic. The banner is otherwise default-on for every command execution path, including `upload`, `upload raw`, `download`, `queue create`, `share create`, `share revoke`, and `share cleanup`; `--version` never calls the formatter at all.

Be deliberate about streams. The banner must never corrupt machine-readable stdout or downloaded bytes. Each handler resolves, via the same corruption-avoidance rules, whether its stdout is human-readable (write the banner to stdout immediately before the command-specific output) or carries a byte stream, JSON document, credentials-only script output, parseable line-oriented output, or another machine-readable contract (write the banner to stderr instead). No current command needs a further exception; if a future command's contract makes even stderr unsuitable for the banner, decide that command's banner routing explicitly when it's introduced rather than in this plan's scope. Cover direct `download` byte output, `--json` output modes, and representative `file-id:`, `share-key:`, `share-url:`, `queue-file:`, `share-revoked:`, and `share-cleanup:` outputs explicitly in tests. Keep progress and error messages on stderr as they are today.

Extract shared terminal capability detection into a CLI-level abstraction, for example under `ShadowDrop.Cli.Terminals`, instead of having banner rendering depend on the existing private detection in `DownloadProgressReporterFactory`. The shared surface should expose the generic inputs already used by download progress selection, such as stderr redirection, CI detection, and rich-output support, through an injectable provider. Update download progress reporting and banner rendering to consume that shared provider so tests can force rich and plain modes without relying on the real process terminal.

When terminal capabilities allow ANSI/rich output, apply color by banner segment rather than by whole line. Use `Spectre.Console` named colors, with `dodgerblue1` encoded as the border color constant and `cyan1` encoded as the highlight color constant in the banner renderer. The outer border and bottom hyphen run should use `dodgerblue1`:

```text
.--                       --.
`-- --------------------- --´
```

The slash markers should use `cyan1`:

```text
   //                   \\
   /                     \
```

The literal `ShadowDrop ` should be white and bold. The literal `v` should be bold and use `dodgerblue1`. The numeric version portion, for example `1.0.0`, should be bold and use `cyan1`, matching the slash markers. Keep a deterministic no-color fallback for redirected output, CI, and tests that do not opt into terminal styling.

Add focused NUnit/FluentAssertions coverage in `tests/ShadowDrop.Cli.Tests/CliApplicationTests.cs` for root `--version`, empty invocation help, accepted `-h`/`--help`, rejected `-?`, the simple help version header, first-line and second-line command banner behavior, second-line padding with a longer version, and global `--no-banner` suppression for command output. Include an assertion that `--version` does not print the decorated banner and that error-only parse, validation, configuration/TLS, and handler-failure paths do not emit the banner. Add renderer-level tests for both deterministic plain output and colored segment/style decisions. Extend command handler tests where command-specific output routing is easier to assert, especially direct download byte output, `--json` output modes, representative parseable script-style outputs, and representative human-readable command outputs. Update the existing `InvokeAsync_ShouldRouteShareRevokeCommand` test in the same file: its exact-match
assertion (`share-revoked:<id>` with no banner) will no
longer hold once the banner is default-on for command execution. Keep test doubles manual per `tests/AGENTS.md`.
