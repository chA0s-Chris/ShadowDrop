# Add one-line install/update scripts for ShadowDrop CLI

## Rationale

Provide supported Linux/macOS and Windows installation scripts that download, verify, and install the correct ShadowDrop CLI release without administrator privileges. Re-running either script must update an existing user-scoped installation, and the documentation must make the one-line commands discoverable.

## Acceptance Criteria

- [x] `install.sh` installs or updates the latest stable CLI on supported Linux and macOS systems through POSIX `/bin/sh`.
- [x] `install.ps1` installs or updates the latest stable CLI on supported Windows systems under Windows PowerShell 5.1 and PowerShell 7.
- [x] Both scripts detect the supported operating system and CPU architecture and select the matching GitHub release asset.
- [x] Both scripts install to a user-scoped default directory, allow an installation-directory override, and replace an existing installed executable.
- [x] Both scripts download and verify the matching asset against the release checksum manifest before installation.
- [x] The one-line installer URLs are served from the repository's `main` branch via `raw.githubusercontent.com`, so they resolve independently of any published release and always deliver the current installer logic.
- [x] Both scripts warn when the target directory is absent from `PATH` and print the installed CLI version.
- [x] README and CLI installation documentation use the stable `raw.githubusercontent.com/chA0s-Chris/ShadowDrop/refs/heads/main/install.sh` and `.../install.ps1` URLs for the one-line install commands and include parameter examples, using the scriptblock invocation form for parameterized Windows installs since plain `iwr ... | iex` cannot pass parameters.
- [x] Bats tests on Linux and macOS and Pester tests under Windows PowerShell 5.1 and PowerShell 7 verify installer behavior against local fixtures, including the documented `curl ... | sh`, `curl ... | sh -s -- <options>`, `iwr ... | iex`, and parameterized `& ([scriptblock]::Create((iwr ...))) <options>` invocation modes.

## Technical Details

Add repository-root `install.sh` and `install.ps1` entry points. Keep the Unix script compatible with POSIX `/bin/sh`, and support both Windows PowerShell 5.1 and PowerShell 7 in the Windows script. Each script resolves the latest stable release through GitHub's version-independent `releases/latest/download/<asset>` redirect URLs, avoiding the GitHub Releases API and any JSON parsing: download `CHECKSUMS.sha256` first — its name carries no version — and treat its entries as the release's asset list. Map the host platform to the existing runtime identifiers: `linux-x64`, `linux-arm64`, `osx-x64`, `osx-arm64`, `win-x64`, and `win-arm64`; fail with an actionable message for unsupported platforms or architectures. Release assets are named `shadowdrop-<version>-<rid>` (with an `.exe` suffix on Windows), so select the manifest entry by its runtime-identifier suffix rather than constructing the file name from an assumed version, then download the binary by the exact name the manifest lists.

Use `~/.local/bin` as the Unix default and `$env:LOCALAPPDATA\ShadowDrop\bin` as the Windows default. On Unix, require `curl`, accept `sha256sum` with `shasum -a 256` as the fallback, and fail with an actionable message when a required tool is unavailable. The PowerShell implementation should use only commands available in both Windows PowerShell 5.1 and PowerShell 7. Create the target directory when needed, stage downloads in a temporary location, validate the selected binary's SHA-256 value against `CHECKSUMS.sha256`, then install it with the correct executable name and permissions. Avoid elevation for these defaults, preserve no stale temporary artifacts, replace prior installations atomically where the platform permits it, warn when the install directory is not represented in the current `PATH`, and execute the installed binary's version command after a successful install.

Expose Unix flags and PowerShell parameters that match the documented `--install-dir`/`-InstallDir` usage. Because plain `iwr ... | iex` cannot receive parameters, document `& ([scriptblock]::Create((iwr -useb <url>))) -InstallDir ...` as the parameterized Windows one-liner and keep `iwr ... | iex` as the defaults-only form. Serve both installer scripts directly from the repository's `main` branch rather than as GitHub release assets: the release workflow stays unchanged, and the one-liners fetch `https://raw.githubusercontent.com/chA0s-Chris/ShadowDrop/refs/heads/main/install.sh` and `https://raw.githubusercontent.com/chA0s-Chris/ShadowDrop/refs/heads/main/install.ps1`, which resolve before the first release ships installers and always deliver installer fixes without waiting for a release. Update `README.md` and `docs/CLI.md` to use these URLs as the primary installation routes while retaining platform, checksum, and PATH guidance.

Add new installer-test jobs to the pull-request CI: Bats on `ubuntu-latest` and `macos-latest`, and Pester on `windows-latest` under both Windows PowerShell 5.1 and PowerShell 7. The existing `ci.yml` runs only on `ubuntu-latest`, so these jobs must be created — in `ci.yml` or a dedicated workflow, at the implementer's discretion. Bats is not preinstalled on GitHub-hosted runners, so the jobs must install it with a pinned version. Structure network and process interactions around local installer, asset, checksum, and installed-binary fixtures so tests can exercise the documented `curl ... | sh`, `curl ... | sh -s -- <options>`, `iwr ... | iex`, and `& ([scriptblock]::Create((iwr ...))) <options>` forms and independently cover platform/architecture mapping, manifest asset selection, checksum validation, argument handling, installation replacement, PATH warnings, version output, and expected failures without downloading a real GitHub release.
