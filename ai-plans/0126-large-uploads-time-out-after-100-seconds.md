## Rationale

Issue [#126](https://github.com/chA0s-Chris/ShadowDrop/issues/126) reports that large CLI uploads restart from zero and ultimately fail when an attempt runs longer than `HttpClient`'s implicit 100-second timeout. The retry path correctly recreates the encrypted multipart request, but uploads are not resumable, so a duration-based timeout wastes all bytes already transferred. Resetting the existing rich progress task also produces a misleading negative transfer speed.

Make the timeout policy intentional for long-running streaming uploads, keep cancellation and bounded transient retries predictable, and make retry progress accurately represent distinct attempts. Document the reverse-proxy settings required to preserve streaming behavior and request-size limits, including how to apply them in Nginx Proxy Manager.

## Acceptance Criteria

- [x] Streaming upload attempts have no total-duration timeout and are not subject to `HttpClient`'s default 100-second overall request timeout.
- [x] Each upload attempt has a fixed, non-configurable 10-minute inactivity timeout that resets whenever encrypted request bytes are successfully written; expiration is treated as a transient transport failure under the existing retry policy.
- [x] Caller cancellation stops the active upload immediately without retrying, while non-streaming control-plane requests retain a 100-second total deadline and streaming download requests remain exempt from any total-duration deadline.
- [x] Retriable upload failures still use the existing bounded attempt count and restart the non-resumable request from byte zero.
- [x] Rich upload progress represents each retry as a new attempt without displaying a negative transfer speed, resets its byte, timer, and speed state, and preserves one clear retry notification and terminal result per file.
- [x] Terminal byte, elapsed-time, and average-speed statistics describe only the final upload attempt.
- [x] If an upload is interrupted before the API receives or claims it, its unclaimed reservation remains subject to the existing one-day expiration and pruning policy.
- [x] If an upload is interrupted after the API claims its reservation, partial blob data is deleted and the claim is released so the reservation can be retried while it remains valid.
- [x] Deployment documentation includes an nginx streaming-upload example that distinguishes request buffering from response buffering and explains temporary-storage and upstream-retry trade-offs.
- [x] Deployment documentation explains multipart request-size headroom, upstream HTTP/1.1 compatibility, inactivity timeout semantics, and limits imposed by additional proxy layers.
- [x] Nginx Proxy Manager guidance identifies the Proxy Host's **Advanced** custom nginx configuration and tells operators to align its values with their ShadowDrop limits.
- [x] Automated tests cover the explicit HTTP timeout policy, long-running upload behavior without a real 100-second delay, timeout versus caller-cancellation classification, bounded retries, retry-safe rich progress, and interrupted reservation/blob cleanup behavior.

## Technical Details

Replace the implicit timeout inherited by `CliHttpClientFactory.CreateClient` with explicit per-request policies. Configure the shared `HttpClient` without a total timeout so an active streaming upload can run for any duration, then retain a 100-second total deadline for non-streaming control-plane requests through request-scoped cancellation rather than `HttpClient.Timeout`. Preserve all existing TLS trust modes and apply the control-plane policy consistently across the CLI clients that share this `HttpClient`. Streaming download requests issued through `CliDownloadSession` share the same `HttpClient` and are likewise exempt from any total-duration deadline; download-specific inactivity handling is out of scope for this plan.

Give each streaming upload attempt a separate 10-minute inactivity deadline beginning when the attempt starts and resetting after each encrypted request-body write reports progress. The deadline continues while the client waits for response headers after the body is sent, so a stalled connection or upstream also terminates deliberately. Keep caller cancellation and inactivity cancellation distinguishable: caller cancellation propagates immediately without retry, whereas inactivity is normalized as a transient transport failure and follows the bounded retry path. Cancellation tokens must continue to flow through request creation, encryption, serialization, retry delays, and response processing. Encapsulate timer and linked-token handling in a small policy abstraction if that keeps the lifecycle and `TimeProvider`-based tests deterministic.

Keep `UploadApiClient.SendWithRetryAsync` at three total attempts and preserve its distinction between caller-requested cancellation and transient network timeout or connection failures. Do not add resumable uploads or cumulative byte accounting across attempts. Make timeout behavior testable through existing handler and delay seams or a small injectable policy abstraction so automated tests do not wait for wall-clock-scale timeouts.

Update `SpectreUploadProgressReporter` so retrying does not move a speed-tracked task backwards from a multi-gigabyte value to zero. Model each attempt independently by replacing or fully reinitializing the task and its byte, timer, and speed state while retaining the per-file description, retry line, and final success/failure output contracts. Terminal byte, elapsed-time, and average-speed statistics describe only the final attempt. Keep plain-text and JSON behavior unchanged.

Verify the current reservation lifecycle rather than introducing a new upload protocol. A reservation created before a buffering proxy receives the full body remains unclaimed and is removed through the repository's existing one-day expiration and pruning policy. A request interrupted after the API claims its reservation must remove partial blob data and release the claim through `UploadPersistenceService`, allowing the same reservation to be retried while it remains valid. Add focused coverage where the current tests do not already prove these two interruption points and their resulting repository and storage states.

Expand the reverse-proxy section in `docs/DEPLOYMENT.md` with a generic checklist and a representative nginx location. Cover `client_max_body_size` for the complete multipart body with headroom aligned to ShadowDrop's effective Kestrel request limit, `proxy_request_buffering off`, explicit `proxy_http_version 1.1` for compatibility with older nginx versions, and finite `client_body_timeout`, `proxy_send_timeout`, and `proxy_read_timeout` values described as inactivity rather than total-duration limits. Explain that disabling request buffering forwards the request to ShadowDrop immediately and avoids client-body temporary-file buffering, but prevents nginx from retrying the partially forwarded request against another upstream. Note that every ingress, CDN, or load balancer in front of nginx may impose separate limits. Include concise Nginx Proxy Manager guidance directing operators to place the server-compatible directives in the Proxy Host's **Advanced** custom nginx configuration and
align example sizes with their own ShadowDrop configuration.
