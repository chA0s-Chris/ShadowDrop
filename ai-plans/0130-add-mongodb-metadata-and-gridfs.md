# Add MongoDB metadata and GridFS persistence providers

> Issue: [#130](https://github.com/chA0s-Chris/ShadowDrop/issues/130)

## Rationale

ShadowDrop currently fixes metadata persistence to LiteDB and encrypted blob persistence to the local filesystem even though its repository and storage interfaces were intended to support replaceable implementations. Introduce MongoDB through [Chaos.Mongo](https://github.com/chA0s-Chris/Chaos.Mongo) as an independently selectable metadata provider and as a GridFS blob provider, while retaining the existing local providers and allowing every metadata/blob combination. Deliver the feature incrementally within one issue and pull request so each persistence slice remains buildable and verifiable. Provider selection must not imply automatic migration of existing data.

## Acceptance Criteria

- [ ] Chaos.Mongo configures and provides MongoDB access whenever either selected persistence provider requires MongoDB.
- [ ] Metadata and blob providers are selected independently through configuration, with existing LiteDB and filesystem behavior retained as the defaults.
- [ ] LiteDB metadata with filesystem blobs, MongoDB metadata with filesystem blobs, LiteDB metadata with MongoDB GridFS blobs, and MongoDB metadata with MongoDB GridFS blobs are all supported.
- [ ] MongoDB implementations persist uploaded-file metadata, share metadata, and admin-token credentials while preserving the existing provider-neutral application contracts.
- [ ] MongoDB metadata operations preserve reservation concurrency, share-token uniqueness, uploaded-file single-use, cleanup state, and admin-token bootstrap guarantees across multiple application instances.
- [ ] A MongoDB GridFS implementation of `IBlobStorage` streams saves and reads, performs idempotent deletion, reports missing blobs through the existing provider-neutral behavior, and aborts failed or cancelled uploads without leaving file or chunk documents when MongoDB remains available for cleanup.
- [ ] Range downloads seek efficiently when the selected blob stream supports seeking and retain buffered skipping as a fallback.
- [ ] MongoDB collections and indexes are initialized through Chaos.Mongo configurators or migrations.
- [ ] Cleanup in MongoDB-backed deployments uses Chaos.Mongo distributed locking for process-level coordination while remaining safe to repeat if a lock lease expires.
- [ ] Configuration validation and filesystem initialization only require and initialize settings used by the selected providers.
- [ ] MongoDB 5.0 and newer standalone servers and replica sets are supported, while sharded-cluster support is not claimed by the initial implementation.
- [ ] Selecting either MongoDB provider verifies connectivity and required initialization during startup and prevents the application from starting when MongoDB is unavailable or initialization fails.
- [ ] Startup diagnostics identify the selected providers and storage state without logging MongoDB credentials or other secret material.
- [ ] Deployment documentation covers MongoDB and GridFS configuration, secret handling, backup and restore considerations, all supported provider combinations, their horizontal-scaling constraints, and the absence of automatic data migration when changing providers.
- [ ] Shared contract tests run against both metadata implementations and both blob implementations, alongside focused unit tests for configuration and dependency registration.
- [ ] The full MongoDB integration suite runs against a concretely pinned newest-stable MongoDB image on both a real standalone server and a replica set, covering concurrency-sensitive behavior and GridFS failure handling.
- [ ] Focused metadata-provider, GridFS-provider, startup, and four-combination smoke tests run against a concretely pinned latest MongoDB 5.0 patch image.
- [ ] Application-level smoke tests cover all four provider combinations while existing local tests remain intact.

## Technical Details

Keep `IUploadedFileMetadataRepository`, `IShareMetadataRepository`, and `IBlobStorage` free of LiteDB, MongoDB, filesystem, and GridFS types. Add stable provider selectors to `MetadataOptions` and `StorageOptions`, defaulting to LiteDB and the local filesystem for backward compatibility. Add MongoDB connection/database options and GridFS bucket options without requiring unused LiteDB paths or filesystem roots. Refactor `ShadowDropOptionsBinding` so it validates provider values first, validates only the active settings, resolves and protects only active local paths, and never emits the MongoDB URL in effective-configuration logging. Support MongoDB 5.0 and newer on standalone servers and replica sets without depending on transactions. Do not claim sharded-cluster support until that topology is explicitly validated.

Update the composition root to register each metadata and blob implementation independently. Call Chaos.Mongo's `AddMongo` whenever MongoDB metadata or GridFS is selected, map metadata document types to stable collection names, and reuse the singleton `IMongoHelper`/connection for both MongoDB providers when both are enabled. Keep provider-specific registrations and initialization out of upload, share, download, and cleanup services. Whenever either MongoDB provider is active, perform a startup connectivity check and run all required configurators or migrations before accepting traffic; an unavailable server or failed initialization must fail startup. Purely local configurations must neither create a MongoDB client nor require MongoDB settings.

Implement the work in four sequential stages within the same issue, branch, and pull request. First establish provider selection, conditional validation, and registration while retaining the local defaults. Next add MongoDB metadata repositories and admin credentials. Then add GridFS storage and seek-aware downloads. Finally add distributed cleanup coordination, deployment documentation, and full provider-matrix verification. Keep every stage buildable and run its focused tests before proceeding so failures remain attributable even though the final pull request is intentionally larger.

Implement MongoDB uploaded-file and share repositories with asynchronous driver operations and cancellation propagation. Model timestamps consistently and preserve the existing record mappings and reservation-retention rules. Reservation claim, release, and completion must use atomic conditional filters and updates rather than process-local locks. Use database constraints for durable invariants, including unique identifiers and share token hashes and a unique multikey index that prevents a file id from being referenced by different share documents; retain request validation for duplicate file ids within one share. Translate duplicate-key and conditional-update outcomes into the same domain behavior exposed by the LiteDB implementations. Run counts, sums, cleanup-candidate selection, and other statistics as server-side MongoDB queries or aggregations rather than loading entire collections.

Move admin credentials behind a persistence-neutral repository so selecting MongoDB metadata does not retain a hidden LiteDB dependency. Make fixed-id bootstrap initialization safe when multiple instances start concurrently: one insert wins and other instances load the resulting credential. During startup, asynchronously initialize or load the selected provider's credential and cache its hash, salt, and iteration count in `AdminTokenService`. Keep request-time `IsValidToken` and the endpoint authorization filter synchronous, hash candidate tokens locally, and compare hashes in constant time so authorization does not perform a database request. When admin operations are disabled, do not initialize the credential repository or require the bootstrap environment variable. Runtime credential rotation and cache invalidation remain outside this issue. MongoDB bootstrap and validation must not log the bootstrap token, stored hash, salt, or connection credentials.

Create Chaos.Mongo configurators or migrations for uploaded-file, share, and admin-token indexes. Include indexes supporting reservation lookup and pruning, file lookup, unique token lookup, single-use file enforcement, cleanup candidate queries, and status/statistics queries. Initialization must be idempotent and safe when multiple API instances start together.

Implement `MongoGridFsBlobStorage` over a stable, configurable `GridFSBucket<Guid>` obtained from the `IMongoDatabase` exposed by Chaos.Mongo. Use the ShadowDrop file id directly as the GridFS file id and return `fileId.ToString("N")` as the canonical GridFS `BlobKey`. Parse only that representation for open and delete operations, reject malformed GridFS keys, and never fall back to filename revision lookup. Continue treating `BlobKey` as opaque outside this provider. A failed save can occur before `UploadPersistenceService` receives an `UploadBlobDescriptor`, so open the upload with its predetermined id and use the driver's abort mechanism when copying fails or is cancelled. When MongoDB remains reachable, cleanup must leave neither a GridFS file document nor chunks for that id. Preserve the original upload exception if abort or additional cleanup also fails; log the residual orphan risk without secret material when an outage prevents complete cleanup. Translate GridFS missing-file
results to the not-found exception contract already consumed by downloads, and keep deletion idempotent.

Preserve `UploadPersistenceService`'s existing compensating workflow instead of treating MongoDB metadata and GridFS as one atomic transaction; GridFS does not support multi-document transactions. Successful GridFS saves must return the actual encrypted byte length so declared-length validation remains provider-independent. Open GridFS download streams as seekable so the existing seek fast-path in the shared skip helper applies; the helper already seeks for seekable streams and retains the read-and-discard loop for non-seekable ones. Add a regression test covering the seek path.

Use Chaos.Mongo distributed locks for workflow-level coordination rather than document invariants. Extend `ShareCleanupRunner.RunIfIdleAsync`, which is already shared by scheduled and manual callers, with optional distributed coordination when MongoDB is configured. Retain its `SemaphoreSlim` as the first, inexpensive same-process guard; after acquiring it, attempt the named distributed cleanup lock and return the existing skipped result when another instance owns that lock. Let MongoDB connection and lock-operation failures propagate as operational failures rather than reporting ordinary overlap. Choose a conservative lease and preserve idempotent blob deletion and conditional cleanup-state updates so a run remains correct if work outlives the lease or an instance terminates. Purely local deployments must continue using the in-process guard without requiring a MongoDB connection. Distributed locking coordinates cleanup work only; it does not make a local metadata or blob provider
safe for horizontal scaling.

Document provider switching as selecting a backend, not moving data: existing LiteDB metadata and filesystem blobs remain in place until an explicit migration facility is implemented separately. Explain that backup and restore must cover the active metadata store and blob store together, including both GridFS bucket collections when GridFS is selected. All four provider combinations support single-instance deployment. MongoDB metadata with GridFS is the standard horizontally scaled configuration. MongoDB metadata with filesystem blobs may use multiple instances only when `LocalRoot` is an appropriately shared filesystem available consistently to every instance. Treat LiteDB metadata combinations as single-instance configurations unless a shared-storage arrangement is separately validated. Document MongoDB 5.0 as the initial minimum, standalone and replica-set support, the unvalidated status of sharded clusters, and the need to re-evaluate the minimum server version whenever
Chaos.Mongo or its MongoDB driver dependency is upgraded.

Add shared contract tests that run against both metadata implementations and both blob implementations, plus focused unit tests for configuration and dependency registration. Resolve the newest stable MongoDB container image available during implementation and the latest available MongoDB 5.0 patch image, pinning both concrete versions rather than using floating tags. Run the full MongoDB integration suite against the newest stable release. Use Testcontainers or the project's equivalent integration fixture to exercise both a real standalone server and replica set, including startup failure, atomic reservation races, concurrent share creation, concurrent admin bootstrap, index creation, distributed cleanup coordination, missing GridFS data, cancellation after chunks have been written, orphan file/chunk cleanup, preservation of the original upload exception, and seekable reads. Against MongoDB 5.0, run focused metadata-provider, GridFS-provider, startup, and four-combination smoke
tests; do not duplicate the expensive concurrency and replica-set suites unless a version-sensitive failure requires it. Removing the MongoDB 5.0 test job later must be accompanied by an explicit change to the documented minimum server version. Add application-level smoke coverage for all four provider combinations while leaving existing local tests intact. Sharded-cluster tests are outside the initial scope.
