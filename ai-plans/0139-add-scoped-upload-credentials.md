# Add scoped upload credentials

> Issue: [#139](https://github.com/chA0s-Chris/ShadowDrop/issues/139)

## Rationale

Separate routine encrypted uploads and share creation from server administration so users and automation can receive a narrowly scoped credential without gaining the ability to revoke arbitrary shares, run cleanup, manage credentials, or call unrelated administrative operations. Preserve existing stored data and the bootstrap admin credential while adding safe credential lifecycle management, ownership isolation, and enforceable upload constraints for both persistence providers; moving upload and share creation off the admin routes is an accepted pre-v1 breaking change.

## Acceptance Criteria

- [x] Add a distinct upload-credential model with one fixed `upload-and-share` capability; do not reuse the bootstrap admin credential or introduce independently assignable scopes in this version.
- [x] Generate upload tokens whose secret contains at least 256 bits of cryptographic randomness and whose selector contains at least 128 bits of cryptographic randomness.
- [x] Store the token secret only as a versioned slow salted hash and compare it in fixed time.
- [x] Use a separate non-exposed lookup digest for efficient authentication.
- [x] Never persist or log plaintext tokens.
- [x] Return a new plaintext upload token exactly once from its authenticated administrative creation operation and make it unrecoverable afterward.
- [x] Support a human-readable name of 1 to 100 characters, optional expiration, optional maximum encrypted file bytes, and optional maximum aggregate encrypted share bytes; defer request-count and byte-budget usage quotas to future work.
- [x] Add admin-authorized API and CLI operations to create, list, inspect, and revoke upload credentials, including `shadowdrop token create`, `shadowdrop token list`, `shadowdrop token inspect <credential-id>`, and `shadowdrop token revoke <credential-id>`.
- [x] Make credential list and inspection bounded and scriptable, expose only the management ID, name, creation time, optional expiration, revocation time, last-used time, fixed capability, and configured constraints, and never expose plaintext tokens, hashes, salts, lookup digests, or the bootstrap admin credential.
- [x] Add scoped upload and share-creation endpoints that accept a valid upload credential, while ensuring that such credentials cannot call share revocation, cleanup, credential management, or any other admin-only endpoint.
- [x] Bind every upload reservation and completed uploaded-file record to the authenticating upload credential, and permit a scoped credential to upload, inspect, or create shares only with reservations and files owned by that same credential.
- [x] Preserve bootstrap-admin authority over upload/share creation and legacy ownerless records, but never let one scoped credential claim ownerless files or files owned by another credential.
- [x] Return only an allow-listed immutable file-metadata projection from `GET /api/uploads/{fileId}`, excluding blob keys, owner IDs, and other persistence internals; return the same `404` response to a scoped caller for missing, ownerless, and foreign-owned records, while allowing the bootstrap admin to inspect any record through the same safe projection.
- [x] Enforce credential expiration and revocation on every authenticated request.
- [x] Apply the effective encrypted-file limit—the smaller of the server's derived maximum file payload and the credential-specific limit when configured—before accepting an oversized upload body, while separately enforcing the server's whole multipart-body limit.
- [x] Enforce a credential's configured aggregate encrypted-share limit from immutable encrypted file lengths before creating a share; treat an omitted limit as no aggregate-share cap and do not add a server-wide aggregate-share setting in this version.
- [x] Record `LastUsedAtUtc` monotonically after successful authentication without allowing concurrent requests to overwrite newer activity or alter revocation state.
- [x] Make credential creation, lookup, last-used updates, and revocation atomic and equivalent across LiteDB and MongoDB, with indexes that avoid scanning all credentials during authentication.
- [x] Keep shares created by a credential available after that credential expires or is revoked; revocation prevents new authenticated operations and does not implicitly revoke existing shares or delete uploaded data.
- [x] Add a nullable upload-exposure setting whose effective value inherits the existing admin-exposure setting when omitted, allowing operators to expose scoped uploads separately without changing the exposure behavior of existing configurations.
- [ ] Preserve existing `--upload-token`, `SHADOWDROP_UPLOAD_TOKEN`, and `uploadToken` resolution for routine uploads.
- [ ] Add dedicated `--admin-token`, `SHADOWDROP_ADMIN_TOKEN`, and config-file `adminToken` configuration required by administrative CLI commands, with no fallback to the upload-token setting.
- [ ] Remove the admin-protected upload/share routes without compatibility aliases and move all CLI upload and share creation to the scoped routes; the bootstrap admin token remains valid on the scoped routes, and documentation describes the route change as a pre-v1 breaking change.
- [x] Return the same generic `401` status and response contract for authentication failures without revealing selector, credential, expiration, or revocation matches in responses, errors, or logs; keep unmatched selectors cheap to reject, run the slow hash for every matched credential before evaluating its lifecycle state, and leave strict timing equalization out of scope.
- [x] Support `--json` for credential-management commands with exactly one documented JSON value on stdout; keep banners, warnings, diagnostics, and migration notices on stderr.
- [ ] Add automated authentication, one-time token disclosure, authorization-boundary, ownership-isolation, expiration, revocation, constraint, concurrency, legacy ownerless-record compatibility, redaction, CLI JSON/stdout, and LiteDB/MongoDB parity tests.
- [ ] Update `README.md`, `docs/CLI.md`, `docs/DEPLOYMENT.md`, `docs/DEPLOYMENT_HARDENING.md`, and `docs/SECURITY_TRADEOFFS.md` with credential provisioning, exposure, migration, constraint, ownership, revocation, and secret-handling guidance.

## Technical Details

Deliver issue #139 as three independently testable, sequential pull requests. Each stage is implemented in its own implementation run on its own branch (`0139a-…`, `0139b-…`, `0139c-…`) and merged via its own pull request; a run ends at its stage boundary rather than continuing into the next stage. The first adds credential persistence, authentication, admin management API and CLI operations, indexes, and provider tests, including the admin-token resolution needed by the new `token` commands. The second adds owner-bound scoped routes, exposure configuration, constraints, and API/provider tests; the legacy admin upload/share routes may coexist temporarily in this intermediate pull request so existing clients remain usable. The third migrates routine CLI operations to the scoped routes, applies the dedicated admin-token setting to existing administrative commands, removes the legacy admin upload/share routes, updates documentation, and adds end-to-end verification. Every pull request
must keep the build and its relevant tests passing; temporary route coexistence is only an implementation-stage measure and the completed issue retains no compatibility aliases.

Add an `UploadCredentialRecord` and `IUploadCredentialRepository` under the API security infrastructure, with LiteDB and MongoDB implementations selected in `DependencyInjection`. Store a random management `CredentialId`, bounded display name, creation/expiration/revocation/last-used timestamps, the fixed capability, nullable encrypted file/share byte limits, a digest of a random token selector, and versioned PBKDF2 salt/hash parameters for an independent high-entropy token secret. The external token should contain the selector and secret, but persistence and administrative projections must expose neither; the management ID is deliberately unrelated to the selector. Extract or share the existing admin-token hashing primitives where useful without changing the persisted bootstrap-admin format. Authenticate through an indexed selector-digest lookup. Reject malformed tokens and unmatched selectors without running the slow hash; when a selector matches a record, always derive and compare
the secret hash in fixed time before combining that result with expiration and revocation state. Every failure returns the same generic `401` response and omits selector, secret, digest, and lifecycle-match details from errors and logs. Uniformity applies to the response and diagnostic contract rather than response time, and strict timing equalization is out of scope. Do not cache mutable credential validity across requests, so revocation is observed consistently by all API instances that share MongoDB.

Add an upload-credential authentication service and endpoint filter that produces an explicit request-scoped authorization context containing the management ID and effective limits. Use a separate upload-or-admin policy only on routine upload/share routes; keep `RequireAdminBearerToken` authoritative for credential management, cleanup, revocation, and other administrative operations. Update `LastUsedAtUtc` with an atomic maximum operation after successful authentication; MongoDB can use a native `$max` update, while the LiteDB implementation may satisfy the same monotonic guarantee with a guarded conditional update. A request already authorized when a concurrent revocation commits may finish, but every authentication begun afterward must fail.

Move normal operations to `GET /api/uploads/capabilities`, `POST /api/uploads/reservations`, `POST /api/uploads`, `GET /api/uploads/{fileId}`, and `POST /api/shares`. Refactor the route mapping out of the parent admin-filtered group so scoped credentials can reach only these handlers. The intermediate scoped-route pull request may retain `/api/admin/uploads` and `/api/admin/shares` while the existing CLI still depends on them; the final pull request removes those operations entirely, keeps no compatibility aliases, and documents the route change as a pre-v1 breaking change. New handlers may accept the bootstrap admin credential when admin operations are enabled, keeping existing admin tokens usable while ensuring an upload credential never gains admin access. `GET /api/uploads/{fileId}` returns an explicit projection containing only `FileId`, `OriginalFileName`, `PlaintextLength`, `EncryptedLength`, `ContentType`, `EncryptionFormatVersion`, `AlgorithmId`, `ChunkSize`, `ChunkCount`,
`KdfSaltBase64`, and `PlaintextSha256`; it never exposes `BlobKey`, `OwnerCredentialId`, or other persistence fields. A scoped caller receives the same `404` response for a missing, ownerless, or foreign-owned record, while the bootstrap admin may inspect any record through that projection. Add `ApiExposureOptions.EnableUploads` as nullable configuration: when omitted, its effective value equals `EnableAdminOperations`; when explicitly set, operators may enable the scoped routes independently. Register upload storage, metadata, credential, and workflow services whenever either uploads or admin operations require them, while initializing the bootstrap admin service only where admin operations are enabled.

Extend upload reservation persistence and `UploadedFileRecord` with an optional owner credential ID. Change `IUploadedFileMetadataRepository.ReserveFileIdAsync`, claim/completion operations, and provider documents so the owner established at reservation cannot be changed during upload. Scoped upload claims must include both file ID and owner in the atomic predicate. Legacy reservations and completed records deserialize with no owner and remain usable only through bootstrap-admin compatibility paths. Pass the authorization context into `UploadPersistenceService`, and reject ownership mismatches with the same safe validation response used for missing or unavailable reservations.

Extend `CreateShareService` to receive the caller authorization context, load the referenced uploaded files, and require every file to have the caller's owner ID before constructing the share. The bootstrap admin path may use both ownerless and credential-owned files for backward compatibility. Record the optional owner credential ID on new `ShareRecord` instances for future administrative projection and audit needs; existing shares deserialize as ownerless. Credential revocation or expiry does not mutate these shares or their files.

The initial constraint set is optional expiration, `MaxEncryptedFileBytes`, and `MaxEncryptedShareBytes`; independently assignable permissions and consumable usage quotas are out of scope. Validate positive constraint values when credentials are created. The effective encrypted-file limit is the smaller of `UploadLimitCalculator.ResolveMaxFilePayloadBytes(Upload.MaxBytes)` and the credential's `MaxEncryptedFileBytes` when configured; continue to enforce `Upload.MaxBytes` separately as the maximum whole multipart request body. Resolve the effective encrypted-file limit before `MultipartUploadRequestReader` consumes the body, and validate both declared and actual encrypted lengths. The effective aggregate-share limit is the credential's `MaxEncryptedShareBytes`; `null` means no aggregate-share cap, and this version adds no server-wide aggregate-share setting. For share creation, sum the immutable `EncryptedLength` values of the distinct owned files with checked arithmetic and reject a
total above a configured effective share limit before persisting the share. The capabilities response returns the effective encrypted-file limit and nullable aggregate-share limit of the authenticated caller.

Add admin-only credential routes below `/api/admin/upload-credentials`: create, bounded newest-first list, inspect by management ID, and idempotent revoke. Use an opaque cursor with a conservative default and maximum page size, and return `404` for unknown management IDs without exposing selector material. Creation returns an allow-listed metadata projection plus the plaintext token; later list/inspect responses use the same projection without the token. Revocation records the first revocation timestamp atomically and does not delete the record, preserving reliable authentication failure and lifecycle inspection.

Extend `CliApplication` with the `token` command group and handler/API-client pairs following the existing command architecture. Routine upload and `share create` continue to resolve `--upload-token`, `SHADOWDROP_UPLOAD_TOKEN`, and config-file `uploadToken`, but use the new scoped routes. Add `--admin-token`, `SHADOWDROP_ADMIN_TOKEN`, and config-file `adminToken` for token management and existing administrative commands such as share revoke and cleanup. Admin commands require the resolved admin token and fail with a clear error pointing at the new configuration when it is absent; they never fall back to the upload-token setting, and upload commands must not fall back from an invalid scoped token to an administrative operation. Keep source-generated JSON contracts in the CLI serialization context, emit the creation secret only to the selected stdout/JSON destination, and avoid repeating it in diagnostics.

Cover the authorization and service behavior primarily through sociable API unit tests with explicit test doubles only at persistence and I/O boundaries. Add provider contract tests for indexed lookup, atomic creation/revocation/last-used behavior, owner-bound reservation claims, backward-compatible ownerless records, and concurrent updates, plus Mongo integration coverage for multi-instance revocation semantics. CLI tests should verify command routing, configuration precedence, one-time secret output, redaction, JSON shape, and stdout/stderr separation. Documentation must state that credential names and management IDs are administrative metadata, token plaintext is displayed once, existing shares survive credential revocation, usage-count quotas are not part of the initial release, and the removal of the admin upload/share routes is a pre-v1 breaking change that requires scripts to adopt the scoped routes and the new admin-token setting.
