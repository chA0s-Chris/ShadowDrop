# Add operator visibility commands

> Issue: [#142](https://github.com/chA0s-Chris/ShadowDrop/issues/142)

## Rationale

Give operators safe, scriptable API and CLI views of server health, compatibility, and share lifecycle state without requiring direct persistence access. Keep public responses minimal, require administrative authorization for operational detail, and make every response an explicit projection that cannot disclose bearer tokens, token hashes, encryption material, storage locations, configuration secrets, or raw failures. This work depends on the scoped upload-credential model from #139 being implemented first; the operator surface must reuse that model rather than treating the existing bootstrap admin token as an upload-scoped credential. After #139 is implemented, reassess this umbrella plan against the resulting credential architecture and break it into focused implementation slices before starting implementation; the slice boundaries are intentionally deferred until that dependency is concrete.

## Acceptance Criteria

- [ ] Add operational protocol version `1` and explicit API contracts for minimal public status, upload-authorized limits, authenticated administrative status, paginated share summaries, and detailed share inspection at the documented routes and with the documented camel-case JSON fields.
- [ ] Add `shadowdrop server status` with concise human-readable output that distinguishes unreachable, live-but-not-ready, ready, unauthorized, and protocol-incompatible outcomes.
- [ ] Add `--verbose` and `--json` status modes with stable, documented field names and reason categories; verbose operational detail requires the admin credential.
- [ ] Use exit code `0` for a healthy compatible result, `2` for unhealthy or not ready, `3` for connectivity or TLS failure, `4` for authorization failure, `5` for protocol incompatibility, and `6` for a not-found result such as an unknown share ID, while retaining `1` for usage and unexpected command failures.
- [ ] Add `shadowdrop share list` with a default page size of `50`, a maximum page size of `200`, deterministic newest-first ordering with a provider-stable share ID tie-breaker, repeatable OR-combined lifecycle filters, opaque query-bound cursors, and exact provider-side matching totals.
- [ ] Add `shadowdrop share inspect <share-id>` with safe lifecycle, cleanup, file-count, and ciphertext-size details; return a stable not-found outcome for unknown IDs.
- [ ] Support `--json` for every new command, emit one documented JSON value on stdout, and keep banners, progress, warnings, and diagnostics on stderr or suppress them as appropriate for machine-readable use.
- [ ] Keep unauthenticated status limited to reachability, liveness, readiness, coarse stable failure categories, protocol compatibility, and non-sensitive capability availability.
- [ ] Require the scoped upload-credential model from #139 and expose effective upload limits and enforced credential constraints only after successful upload-credential authentication.
- [ ] Restrict exact build version, provider names, storage totals, share and cleanup counts, dependency detail, resumable-session counts, and configuration warning codes to the admin credential.
- [ ] Never expose share or download bearer tokens, encryption keys, upload or admin credential plaintext or hashes, token lookup identifiers, connection strings, database hosts, blob keys, filesystem paths, raw configuration values, or internal exception messages.
- [ ] Omit filenames and credential identifiers from summary output. Redact filenames from share inspection by default and allow an explicit `--include-filenames` view only through the admin-authorized endpoint, with the sensitivity documented.
- [ ] Persist only the latest completed cleanup attempt timestamp and its bounded set of sanitized failure categories, replace that outcome on retry, leave cancelled attempts unrecorded, and never store or return raw cleanup exceptions.
- [ ] Avoid unbounded scans and N+1 metadata reads by adding bounded provider-level queries, batch file-size lookups, and provider-appropriate indexes or aggregations for LiteDB and MongoDB.
- [ ] Provide equivalent contracts, filtering, ordering, pagination, totals, and failure semantics for LiteDB/filesystem and MongoDB/GridFS deployments.
- [ ] Probe each configured physical dependency through an explicit operational probe with a timeout defaulting to 5 seconds and caller cancellation so a failing MongoDB or storage dependency cannot hang `server status`.
- [ ] Emit stable structured audit events for administrative status, share-list, and share-inspection outcomes with only the operation, outcome, HTTP status, and elapsed time; never serialize returned records, filters, cursors, filenames, identifiers, credentials, request bodies, exception text, or other secret-bearing data into those events.
- [ ] Add automated authorization, redaction, filename opt-in, pagination, filtering, deterministic-ordering, JSON-contract, stdout/stderr, exit-code, timeout, provider-parity, and degraded-dependency tests.
- [ ] Update `README.md`, `docs/CLI.md`, `docs/DEPLOYMENT.md`, and `docs/SECURITY_TRADEOFFS.md` with command examples, status and exit-code contracts, credential requirements, field sensitivity, and pagination behavior.

## Technical Details

Issue #139 is a hard prerequisite. Reuse its upload-credential authorization policy for the upload-authorized projection; public and download credentials must never receive that projection, and scoped upload credentials must never authorize an administrative endpoint. Introduce explicit response records for the operational surface rather than returning persistence records or anonymous objects. Keep the existing `/health/live` and `/health/ready` endpoints as minimal orchestrator probes. Put shared protocol constants and source-generated JSON contracts in `ShadowDrop.Shared` where both the API and Native AOT CLI can use them.

Operational protocol version `1` has a CLI-supported inclusive range of `[1, 1]`. `GET /api/status` is public, `GET /api/status/upload` requires a valid #139 upload credential, and `GET /api/admin/status`, `GET /api/admin/shares`, and `GET /api/admin/shares/{shareId}` extend the `AdminEndpoints` route group and require `RequireAdminBearerToken`. `GET /api/admin/shares/{shareId}` accepts `includeFilenames=false` by default; only the authenticated admin route may set it to `true`. The response contracts are:

| Contract         | Required camel-case JSON fields                                                                                                                                                                                                                         |
|------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Public status    | `protocolVersion` (integer), `live` (Boolean), `ready` (Boolean), `reason` (string), and `capabilities` containing the Boolean fields `publicDownloads`, `adminOperations`, `resumableDownloads`, and `scopedUploads`                                   |
| Upload status    | All public-status fields plus `effectiveLimits`, containing nullable `maxFileBytes`, `maxShareBytes`, `remainingUses`, and `expiresAtUtc` fields; `null` means the authenticated credential has no such constraint                                      |
| Admin status     | All public-status fields plus `buildVersion`, `uptimeSeconds`, `components`, `providers`, `storage`, `shares`, `cleanup`, `resumableSessions`, and `configurationWarnings`                                                                              |
| Share page       | `items`, `nextCursor`, and `totalMatching`; each summary item contains `shareId`, `createdAtUtc`, `expiresAtUtc`, `revokedAtUtc`, `statuses`, `cleanupState`, `lastCleanupAttemptAtUtc`, `cleanupFailureCategories`, `fileCount`, and `ciphertextBytes` |
| Share inspection | The summary fields plus `files`; each file contains `fileId`, `ciphertextBytes`, `originalFilename`, and `displayName`, with both filename fields `null` unless the admin-authorized filename opt-in was requested                                      |

In upload status, byte and usage limits are nullable 64-bit integers and `expiresAtUtc` is a nullable timestamp. In admin status, `buildVersion` is a string and `uptimeSeconds` is a 64-bit integer. `components` is an array whose entries contain string `name`, `state`, and `reason` fields. `providers` contains string `metadata` and `storage` fields; `storage` contains nullable 64-bit `completedFileCount` and `ciphertextBytes` fields; `shares` contains 64-bit `active`, `expired`, `revoked`, `cleanupPending`, `cleanupFailed`, and `cleanupCompleted` fields; `cleanup` contains nullable timestamp `lastRunAtUtc` and a required string `lastOutcome` whose initial value is `not-run`; `resumableSessions` contains nullable 64-bit `activeCount`; and `configurationWarnings` is an array of stable strings. Unsupported or disabled totals are `null`, never fabricated as zero, and the corresponding capability or component reports that it is unavailable.

Share and file identifiers are lower-case UUID `D` strings. `createdAtUtc` and `expiresAtUtc` are required timestamps; `revokedAtUtc` and `lastCleanupAttemptAtUtc` are nullable timestamps. `statuses` and `cleanupFailureCategories` are arrays of the stable strings defined by this plan, `cleanupState` is `pending`, `failed`, or `completed`, `fileCount` is an integer, and ciphertext sizes and `totalMatching` are 64-bit integers. `nextCursor` and the two filename fields are nullable strings.

Public `reason` is always present and is one of `none`, `dependency-unavailable`, `dependency-timeout`, or `configuration-invalid`. Component `state` is one of `ready`, `not-ready`, or `not-applicable`, and its `reason` uses the public set plus `capability-disabled`. JSON arrays are emitted as empty arrays rather than `null`; only fields explicitly described as nullable may be `null`, and fields are not conditionally omitted. Timestamps use UTC ISO 8601 round-trip strings. Non-status errors use a single `{ "reason": string }` contract. Invalid requests return `400` with `invalid-request`, missing or invalid credentials return `401` with `unauthorized`, and an unknown share returns `404` with `not-found`; a malformed or query-mismatched cursor specializes the `400` reason to `invalid-cursor`. A public or administrative status response uses `200` when ready and `503` when not ready while retaining the same response body shape.

The server publishes the single integer protocol version and the CLI compares it with its supported inclusive range rather than requiring identical product build versions. Clients ignore unknown additive JSON fields. Additive fields do not bump the protocol version; removing or changing fields, endpoint shapes, nullability, reason semantics, or filter/cursor semantics does. Exact product/build version belongs only in the admin projection; the public projection exposes only compatibility information needed by the CLI.

Introduce `IOperationalDependencyProbe`, which returns an allow-listed component name, state, and reason without an exception or configuration value. Register probes for the configured metadata and blob-storage dependencies and aggregate them through an evolved structured `IReadinessCheck`. A physical MongoDB dependency shared by metadata and GridFS is pinged once and its result is projected onto both affected logical components; LiteDB uses a bounded metadata operation, and filesystem storage uses a non-destructive root accessibility check. Disabled capabilities report `not-applicable` instead of zero-valued statistics. Give each physical probe its own linked cancellation deadline, defaulting to 5 seconds: deadline expiry maps to `dependency-timeout`, provider failure maps to `dependency-unavailable`, and caller cancellation propagates instead of becoming a health result. Preserve the current minimal public behavior of `/health/live` and `/health/ready` while exposing component
detail only through admin status.

Aggregate operational status through a dedicated service registered by `DependencyInjection`; it should use `ShadowDropOptions`, a process-start timestamp based on `TimeProvider`, `IUploadedFileMetadataRepository` storage and active-reservation statistics, `IShareMetadataRepository` status counts, cleanup-run state, and the dependency-probe results. Public output collapses component failures to the defined coarse categories, while the admin projection identifies only the allow-listed logical component.

Add a query model and cursor page result to `IShareMetadataRepository`, implemented by `LiteDbShareMetadataRepository` and `MongoShareMetadataRepository`. Capture one `nowUtc` for each query. `active` means not revoked and expiring after `nowUtc`; `expired` means expiring at or before `nowUtc`; `revoked` means a revocation timestamp is present; `cleanup-pending` means cleanup is pending and the share is expired or revoked; and `cleanup-failed` and `cleanup-completed` are exact cleanup-state matches. These are independent predicates, so a share may have more than one status. Repeatable status filters combine with OR, no filter returns all shares, and matching totals count each share once.

Use a default page size of `50` and a maximum of `200`; reject non-positive, oversized, or otherwise invalid sizes with `400` and `invalid-request`. Order inside each provider by descending `(CreatedAtUtc, ShareId)`, comparing the tie-breaker as the canonical lower-case UUID `D` string with ordinal semantics so LiteDB and MongoDB agree. Continue strictly after the cursor's sort position. Encode the protocol version, sort position, and normalized filter set in the opaque cursor; reject malformed cursors or cursors used with different filters with `400` and `invalid-cursor`. Return `items`, `nextCursor`, and an exact provider-side `totalMatching`; document that totals can change between requests as records change.

Add the indexes needed to support timestamp, cleanup, expiration, revocation, and provider-stable tie-breaker predicates while preserving backwards compatibility with existing records. Extend `IUploadedFileMetadataRepository` with a bounded batch lookup or equivalent safe projection so list and inspection services can calculate stored ciphertext totals without one query per file. Keep persistence records internal to the query service and map them into allow-listed summary/detail DTOs that omit share-token hashes, download-token data, blob keys, cryptographic metadata, plaintext hashes, and filenames unless the admin-only filename opt-in was requested.

Extend the cleanup state model with `LastCleanupAttemptAtUtc` and a bounded set of `CleanupFailureCategory` values from the latest completed attempt. The initial stable categories are `upload-metadata-missing`, `metadata-unavailable`, `blob-delete-failed`, and `unknown`. `ShareCleanupService` captures one attempt timestamp from `TimeProvider` immediately before processing a share, collects each distinct sanitized category, and atomically replaces cleanup state, timestamp, and categories through both repository implementations after processing finishes. A successful attempt stores `Completed` with an empty category set; a failed attempt stores `Failed` with one or more categories; and a later retry replaces the prior outcome rather than accumulating history. Existing records deserialize with a null timestamp and an empty category set.

Caller cancellation propagates without recording a failure outcome; already deleted blobs remain safe to process idempotently on retry. If the final state write itself fails, emit a sanitized operational log but do not claim that a category was persisted. Do not persist raw exceptions. Track only the latest sanitized cleanup outcome needed for visibility; this feature is not a general event history or metrics system.

Add `server`, `share list`, and `share inspect` command nodes and dispatch paths in `CliApplication`, following the existing handler/API-client split used by the share management commands. Add a dedicated admin-token configuration input rather than treating future scoped upload credentials as administrators, while preserving the existing upload-token path for upload-authorized status fields. Handlers should use the shared TLS factory, bounded control-plane requests, cancellation, source-generated JSON serialization, and replaceable collaborators in `CliApplicationServices`. Human output should be concise and deterministic; JSON handlers should write exactly one contract value to stdout and route all diagnostics to stderr. Map HTTP/TLS, readiness, authorization, compatibility, and not-found results centrally so all commands apply the documented exit-code policy consistently.

Use structured application logs as the audit mechanism rather than adding a separate audit subsystem. Audit the three admin API operations with stable operation names `admin-status-view`, `admin-share-list`, and `admin-share-inspect`, and stable outcomes `success`, `unauthorized`, `not-found`, `invalid-request`, `cancelled`, and `failure`. Include only operation, outcome, HTTP status, and elapsed time. Emit successful and not-found events at information level, unauthorized and invalid-request events at warning level, and failure events at error level; cancellation remains an information event. The authorization filter or surrounding middleware must emit the unauthorized outcome without recording the supplied credential. Do not attach exception text or include identifiers, filters, cursors, filename opt-in values, response records, request bodies, or credentials in an audit event.

Cover endpoint authorization and projections in `ShadowDrop.Api.Tests`, CLI rendering and exit behavior in `ShadowDrop.Cli.Tests`, and provider query/index parity in the existing LiteDB and MongoDB integration fixtures. Include large-enough fixtures to prove page bounds, cursor stability and rejection, OR-combined filters, provider-stable UUID ordering, exact matching totals, batch size aggregation, redaction, backward-compatible cleanup records, multi-category cleanup failures, cancelled attempts, and timeout behavior without adding expensive unbounded test scans. Verify every audit operation/outcome and assert that prohibited values are absent from captured events. Update the operator documentation with the exact field matrix, supported reason strings, exit codes, admin-token configuration, JSON examples, and the security implications of explicitly requesting filenames.
